# RFC Writer Skill

This skill guides the creation of Product Management RFCs at {{COMPANY}}. It ensures alignment with the company's "Fact-Based" culture and specific writing style.

## Core Mandates

1.  **Audience**: The primary audience is stakeholders (other PMs, Engineering Leads, Design, Data), but the perspective is **Product**, not Engineering implementation details (unless specified in a technical subsection).
2.  **Tone**: Direct, objective, and data-driven. No corporate fluff.
3.  **Visuals**: "A picture is worth a thousand words." Always suggest or include diagrams (Mermaid, Excalidraw, or image placeholders) for complex concepts.
4.  **Style Guide Adherence**: You **MUST** invoke the `pm-writing-standards` skill, which inlines the four axioms (Economy, Structure, Precision, Calibration) and the final-pass checklist. Do not generate content without validating it against the rules loaded by that skill.

## RFC Structure

The RFC must follow the immutable structure defined in `references/rfc-template.md`.
**Do not invent new top-level sections.**

The structure is immutable; the content distribution within it is the writer's call. Sections reference each other by design — a fact established in the lead does not need to be restated in Context, even if Context is a required section. Treat each section as a place to put content the earlier sections did not cover, not as a quota to fill. A templated section can legitimately be brief, lean on cross-references back to earlier sections, or focus on the subset of the topic that has not been covered yet. The failure mode this prevents: treating the immutable structure as a mandate to restate the same fact in multiple sections, producing a doc that insults the reader's time by repeating information across required slots.

### Key Sections Guidance

-   **Status**: Draft/Submitted/Accepted.
-   **What is the problem**: The "Why". Use facts, numbers, and "why now".
-   **Goals**: Success definitions. No "nice to haves".
-   **Non-goals**: What is explicitly out of scope.
-   **Proposed Solution**: How goals are achieved.
    -   *PM Focus*: User journey, value proposition, impact.
    -   *Diagrams*: Mandatory for flows/architecture.
-   **Risks & Considerations**: Drawbacks and countermeasures. Be honest to earn trust.
-   **Dependencies**: Blockers or required efforts from other teams.
-   **Alternatives**: What else was considered and why it was discarded.

## Workflow

1.  **Context Loading**: Read relevant strategy/research docs from `{{KNOWLEDGE_BASE_PATH}}/`.
2.  **Style Loading**: Invoke the `pm-writing-standards` skill to load the writing constraints. All four axioms apply: economy, structure, precision, and calibration.
3.  **Audience Declaration**: Before drafting, name the primary, secondary, and tertiary audiences explicitly. State this in the RFC's metadata table (Reviewers field) and hold it internally for the strict-reader pass in step 5. The default {{COMPANY}} RFC audience is mostly non-engineers (PMs, Engineering Leads outside the system's domain, Design, Data, leadership); only one or two readers will recognise the system's deepest implementation terms.
4.  **Section-by-Section Drafting**: Draft the LEAD first (the opening of "What is the problem"), then PAUSE for user review before drafting the rest. Most failure modes (tone, jargon, dangling claims, audience mismatch) compound across sections; correcting them at section 1 is cheaper than refactoring 200 lines. Only after lead approval, draft Goals → Non-goals → Proposed Solution → remaining sections.
5.  **Pre-flight Review** (run after the full draft is in place, BEFORE declaring done; address findings inline):
    - **Strict-reader pass.** Re-read as the LEAST-technical named audience from step 3. Mark every sentence that audience cannot parse without prior knowledge. Each marked sentence either gets an inline definition, moves to a footnote, moves to the Glossary, or relocates to the Technical Appendix. Engineering specifics (formulas, code identifiers, file paths, complexity notation) belong in the Technical Appendix. Scan for technical adjectives, not just named domain terms. Adjectives like *concentric, stateful, stateless, asymptotic, monotonic, stochastic, idempotent, sharded, cached, indexed, tokenised* read as precise but require domain knowledge to parse. Each marked adjective either gets an inline gloss in plain terms, replacement with descriptive prose, or relocation to the Technical Appendix. The strict-reader test is: would the audience hear *concentric* and know what shape that implies for a candidate set? If not, replace.
    - **Tone calibration pass.** Scan for catastrophic verbs (`burn`, `hurt`, `lock out`, `stall`, `block`, `cripple`). For each, apply the literal-truth check: is the system actually failing in this way, or has it operated this way for years? If years, reframe as opportunity cost or maintenance debt per axiom_4_calibration.
    - **Mechanism-before-name pass.** For each named domain term (DS, Q2C, LTR, geofilt, etc.), confirm the FIRST use cashes out what the thing does, with a concrete example, before any sentence treats it as a known noun. Naming-without-mechanism is a cold-reader-test failure even when the term is later defined in a glossary.
    - **Dangling-claim sweep.** For each header sentence and section-lead sentence, identify the 1–3 sentences that justify it. If they don't exist, either justify inline or delete the framing. "X doesn't fit Y" / "The system has worked for years" / "Three pieces are misaligned" are common dangling patterns.
    - **Acronym discipline.** Every acronym is defined inline on first use within the body. The Glossary at the end is a safety net, not the primary definition.
    - **Definition-vehicle placement (post-structural-edit re-scan).** Footnote anchors, inline parentheticals, and glossary cross-references attach to the FIRST body use of the term, not to a later thematic restatement. This is not a one-time end-of-draft pass — it MUST be re-run after every structural edit (paragraph rewrite, bullet reorder, section move) because rewrites routinely move a term's first body use to a new location while leaving the original definition vehicle anchored downstream. Each time a term's first use moves earlier through a rewrite, re-anchor the footnote/inline gloss to the new first use AND remove downstream redundant vehicles. The recurring failure mode: drafting introduces a term in a "Why now" enumeration with a footnote, a later rewrite promotes the term to the lead, and the footnote stays where it was — second use.
    - **Could-we-continue acknowledgment.** For each pain bullet, confirm the bullet either acknowledges the could-continue status quo explicitly OR represents a genuinely new/escalating failure. A pain bullet on a multi-year-stable system without this acknowledgment is uncalibrated.
    - **User-first framing pass.** For each problem-statement bullet (lead, Impact of inaction, Risk), confirm the header sentence centers the user-facing first-order harm rather than the team-facing or measurement-facing consequence. If the bullet has both kinds of impact, the user-facing one comes first; the metric/operational consequence is positioned as downstream. The "team-first" default is most likely when the source material (dashboards, technical specs, ops docs) is itself team-facing — invert explicitly.
    - **Adjacent-workstream inflation pass.** For each downstream consequence in a problem bullet that names another team's initiative (the X revamp, the Y migration, the Z plan), apply the substitution test: "would I write this consequence if the workstream had a different name?" If the consequence is genuinely material on its own — buyers see degraded discovery, the system blocks future work — keep it and verify the causal trace to the bullet's first-order harm is named. If the consequence reads as a name-drop to bulk up importance, CUT it. Workstream-name-dropping is a calibration failure on the importance axis: it inflates the case past the bullet's actual stakes.
    - **Mechanism-implying-adjective check.** For each adjective in a problem framing that asserts a derivation, curation, or update method (*hand-tuned, learned, automated, manually-curated, hard-coded, dynamic, derived, computed, inferred, heuristic, rule-based, data-driven*), confirm you have evidence for that method. If you cannot point to the source (a code pointer, a Confluence page, a research note), REPLACE with a neutral descriptor that names the artefact without claiming a method (*per-keyword radius* not *hand-tuned radius*; *51,666-keyword list* not *manually-curated list*). Mechanism-implying adjectives are the failure mode where a confident-sounding header survives unchanged into final drafts because nothing in the body contradicts it; the precision check is the antidote.
6.  **User review checkpoint**: Surface the lead and pre-flight findings to the user before drafting the implementation/technical sections. Do not commit the full RFC until the user has approved the lead's framing.
7.  **Post-edit self-pre-flight (after EVERY user-driven rewrite, not just at end-of-draft)**: Whenever the user requests a rewrite to a paragraph, bullet, or section — including small edits to fix a flagged failure — re-run the relevant pre-flight passes (step 5) on the rewritten text AND on the immediately-adjacent text BEFORE handing back to the user. The failure mode this prevents is mechanism-shelfware: pre-flight passes that exist in the skill but never run on rewrites because they were treated as a one-time end-of-draft step. Concretely, after fixing a paragraph for a flagged issue, scan the same paragraph and its neighbours for: (a) the same failure repeated elsewhere; (b) sibling failures the new content introduces (e.g., a rewrite that promotes a term to first use without re-anchoring its footnote); (c) the failure modes most adjacent to the one just fixed (mechanism-implying adjectives → technical adjectives → reader orientation form one cluster; calibration verbs → could-we-continue → cost-naming-traces-to-signal → adjacent-workstream inflation form another). The implicit contract: if the skill defines a pass, the writer runs it after every rewrite, not only at the end.

    **Critical: the fix loop is the danger zone for cross-category regressions.** When the user flags a problem and the writer produces a fix under time pressure, the fix can introduce a new failure in a *different category* than the one being addressed. Run pre-flight against ALL categories on every fix, not just the one being fixed. Especially: any fix that replaces vague/jargon-y language with something more confident-sounding must be checked against the org-reality verification ACTION (`axiom_3_precision`) before being submitted. Replacing "Hybrid Search compatibility window" with "Hybrid Search lands in 2026" fixes the metaphorical-jargon failure but introduces an org-reality overclaim — a different failure category. Default to ASKING the user via AskUserQuestion when a fix needs to assert any claim about organizational reality (timing, funding, priority, ownership, dependency) the writer cannot independently verify; do not pick a confident-sounding alternative under fix-loop pressure.
